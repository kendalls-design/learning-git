01

Repository (repo)

What it does

A repository is the project folder tracked by Git. It contains your source code, a complete history of every change, and configuration that tells Git how to behave. When hosted on GitHub (or GitLab, Bitbucket), it becomes the shared source of truth your team collaborates around.

Considerations

  • Initialize with git init for new projects, or clone an existing remote repo.
  • Keep secrets out of the repo — use .gitignore and environment variables.
  • A repo can have multiple remotes (e.g. origin on GitHub, upstream for open source).
  • README, license, and contributing guidelines help collaborators onboard quickly.

02

Clone

What it does

Cloning downloads a complete copy of a remote repository — including its full commit history, branches, and tags — to your local machine. It also sets up a remote called origin pointing back to the source, so push and pull work out of the box.

Considerations

  • Use HTTPS for simplicity, or SSH if you have keys configured.
  • Large repos with long histories can take time; consider shallow clones (--depth 1) for CI.
  • Cloning is a one-time setup step — after that, use pull to stay up to date.
  • Fork + clone is the standard open-source contribution workflow.

Resources

03

Branch

What it does

A branch is a parallel timeline of development. It lets you work on features, fixes, or experiments in isolation without affecting the main codebase. Git branches are lightweight pointers to commits — creating one is nearly instant.

Considerations

  • Name branches descriptively: feature/login-form, fix/nav-bug.
  • Keep branches short-lived — merge or delete them once work is done.
  • Protect your main branch with required reviews and CI checks.
  • Delete merged branches to keep the repo tidy (git branch -d).

Resources

04

Commit

What it does

A commit is a snapshot of your project at a point in time, paired with a message describing what changed and why. Commits form an immutable chain — each one references its parent — creating a full, auditable history you can travel through.

Considerations

  • Write clear messages: imperative mood, ~50 char subject, optional body for context.
  • Commit early and often — small, focused commits are easier to review and revert.
  • Stage intentionally with git add -p to keep unrelated changes separate.
  • Amending and rebasing rewrite history — avoid on shared branches.

Resources

05

Push / Pull

What it does

Push uploads your local commits to the remote repository, making them available to collaborators. Pull downloads new commits from the remote and merges them into your current branch, keeping your local copy up to date.

Considerations

  • Always pull before starting new work to avoid unnecessary conflicts.
  • Push only commits you've tested — CI on the remote will catch issues too.
  • git pull --rebase keeps history linear; plain pull creates merge commits.
  • If push is rejected, someone else pushed first — pull, resolve, then push again.

Resources

  • git push — official docs · Git
  • git pull — official docs · Git

06

Diff

What it does

A diff shows the visual difference between two versions of code — line by line. It highlights additions (green), deletions (red), and unchanged context. Diffs appear in commits, pull requests, and the output of git diff.

Considerations

  • Review diffs before committing to catch debug code, typos, and accidental changes.
  • Keep diffs small — large PRs are harder to review and more likely to introduce bugs.
  • Whitespace-only diffs can be hidden with -w to focus on logic changes.
  • IDE diff tools and GitHub's PR view make side-by-side comparison easy.

Resources

  • git diff — official docs · Git
  • Pro Git — Viewing History · official book

07

Merge

What it does

Merging combines the changes from one branch into another. Git finds the common ancestor, applies both sets of changes, and creates a merge commit (unless fast-forwarding). This integrates feature work back into main.

Considerations

  • Fast-forward merges happen when main hasn't moved — no merge commit needed.
  • Merge commits preserve the true history of parallel development.
  • Always run tests after merging locally before pushing.
  • On GitHub, merging a PR is the most common way teams integrate work.

Resources

  • git merge — official docs · Git

08

Rebase

What it does

Rebasing replays your branch's commits on top of the latest main, as if you had started from that point. It produces a linear history without merge commits. Think of it as moving your work to the tip of main.

Considerations

  • Never rebase commits that have already been pushed and shared with others.
  • Rebase locally before opening a PR for a cleaner review history.
  • Interactive rebase (-i) lets you squash, reorder, or edit commits.
  • Conflicts can occur during rebase — resolve them commit by commit.

Resources

  • Pro Git — Rebasing · official book

09

Merge Conflict

What it does

A merge conflict occurs when two branches modify the same lines of a file and Git cannot automatically decide which version to keep. Git pauses the merge and marks the conflicting sections — you choose the correct result, then commit.

Considerations

  • Conflicts are normal in collaborative work — don't panic.
  • Look for <<<<<<<, =======, and >>>>>>> markers in files.
  • Use your IDE's merge tool or git mergetool for visual resolution.
  • Communicate with teammates when conflicts involve the same feature area.

Resources

10

Pull Request (PR)

What it does

A pull request is a request to merge your branch into another — typically main. It provides a forum for code review, discussion, automated CI checks, and approval before changes land in the shared codebase. "Please review and merge my branch."

Considerations

  • Write a clear PR description: what changed, why, and how to test it.
  • Link related issues with "Fixes #123" to auto-close them on merge.
  • Keep PRs focused — one feature or fix per request.
  • Respond to review feedback promptly; push new commits to the same branch.

Resources

11

Issue

What it does

An issue is a ticket for tracking bugs, feature requests, tasks, or questions within a repository. Issues provide a structured way to discuss work, assign owners, apply labels, and link to pull requests that resolve them.

Considerations

  • Use labels (bug, enhancement, documentation) to categorize and filter.
  • Write reproducible bug reports: steps, expected vs actual, environment.
  • Link issues to PRs so they auto-close when the fix merges.
  • Issue templates help contributors provide the right information upfront.

Resources
